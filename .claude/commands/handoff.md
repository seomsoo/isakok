현재 작업 상태를 docs/STATUS.md에 업데이트해줘.

## ⚠️ 구조 규칙 — STATUS는 "라이브 + 단계 인덱스", 디테일은 정본 문서

STATUS.md는 **세션 시작 시 한눈에 읽는 라이브 문서**다. 단계별 빌드 디테일을 STATUS에 누적하지 않는다(과거 1100줄+로 비대해져 2026-06-05 인덱스+정본 링크 구조로 분리).

**디테일의 정본(source of truth):**

- 단계 구현 디테일(컴포넌트·훅·서비스·마이그레이션·Codex 수정 이력) → 해당 `docs/specs/{단계}.md` + `{단계}-verify.md`
- 설계 의사결정(ADR) → `docs/ADR.md`
- UI/UX 폴리싱 → `docs/UI-POLISH.md`

**STATUS.md "완료된 것"** = 단계별 **1~2줄 인덱스 + 정본 링크**만. (예: `- **6단계 집 상태 기록** ✅ — 한 줄 요약. 📄 specs/06-property-photo.md(+verify)`)

### 절대 보존(손실 금지)

- "완료된 것" 인덱스에서 **단계를 누락하지 말 것** — 완료 단계는 반드시 1줄 존재. 정본 문서가 없는 단계(예: 하네스 실동작 검증, 문서 스왑)는 **인덱스 줄에 핵심 요약을 그대로 남긴다**(인덱스가 곧 정본).
- "실패한 접근 (반복 금지)" 섹션은 **STATUS에 누적 보존** — 한 번 추가된 항목은 영구 보존(횡단 안티패턴이라 정본이 STATUS 자신).

> Why: 다음 세션 진입 시 STATUS를 빠르게 훑고, 필요하면 인덱스의 링크로 정본 문서를 연다. git log/blame 회피 의도는 유지 — 회상처가 STATUS 본문 → 링크된 specs/ADR로 이동했을 뿐.

## 갱신 방식

1. 기존 STATUS.md를 Read로 읽는다.
2. 이번 세션 디테일은 **정본 문서**(spec/verify·DECISIONS·UI-POLISH)에 기록(없으면 거기에 추가).
3. STATUS "완료된 것" 인덱스에 이번 완료 단계 **1줄 추가/갱신**(기존 줄 보존, 단계 누락 금지).
4. 라이브 섹션만 갱신: "현재 단계", "진행 중인 것", "다음 할 것", "알려진 문제".
5. "실패한 접근"은 누적(이전 항목 보존 + 이번에 새로 배운 것 추가).

## 기록할 내용

1. **현재 단계**: 몇 단계 작업 중인지 + 상태(진행중 / 구현완료-검증대기 / 검증완료 / 다음단계대기)
2. **진행 중인 것**: 아직 끝나지 않은 작업 (있으면)
3. **다음 할 것**: 구체적인 다음 액션 1~3개
4. **알려진 문제**: 발견했지만 아직 안 고친 것 (있으면)
5. **완료된 것(인덱스)**: 이번 완료 단계 1줄 추가 + 정본 링크. 디테일은 정본 문서에.
6. **실패한 접근**: 누적 — 이전 항목 보존 + 이번 세션 신규만 추가

## 형식

```md
# 프로젝트 상태

> 마지막 업데이트: {오늘 날짜}

## 현재 단계

{N}단계: {단계명} — {상태}

## 진행 중인 것

- {내용} (없으면 "없음")

## 다음 할 것

1. {가장 먼저 할 것}
2. {그 다음}

## 완료된 것 (요약 인덱스 — 상세는 각 단계 spec/verify + DECISIONS)

- **{N}단계 {단계명}** ✅ — {한 줄 요약}. 📄 `specs/{파일}.md`(+verify) · ADR-...

## 알려진 문제

- {내용} (없으면 "없음")

## 실패한 접근 (반복 금지)

- {내용} (없으면 "없음")
```
