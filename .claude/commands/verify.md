현재 단계의 스펙 파일을 찾아서 검증을 수행해줘.

## 검증 순서

1. docs/STATUS.md에서 현재 단계 번호를 확인
2. docs/specs/{현재단계 스펙}.md의 "완료 확인 기준" 체크리스트를 하나씩 실제로 확인
   - 빌드 확인: pnpm build 실행
   - 린트 확인: pnpm lint 실행
   - 테스트 확인: pnpm test 실행
   - 나머지 항목은 파일 존재 여부, import 동작 등 직접 확인
3. 스펙에 있는데 구현 안 된 것 → "누락" 목록
4. 구현했는데 스펙에 없는 것 → "스코프 크립" 목록
5. 코드 컨벤션 위반 (CLAUDE.md 규칙 대비) → "컨벤션 위반" 목록
6. **spec-reviewer 자동 호출**: 복잡한 단계(DB + Edge Function + 클라이언트를 함께 건드리는 단계, 또는 스펙 파일이 500줄 이상인 단계)에서는 `spec-reviewer` 서브에이전트를 반드시 호출하여 스펙 vs 구현 심층 비교를 수행한다. 결과를 verify 리포트의 "spec-reviewer 결과" 섹션에 기록한다.
7. **서브에이전트 자동 호출**: `git diff main --name-only`로 변경 파일 목록을 확인하고, 아래 매핑 테이블에 해당하는 서브에이전트를 **병렬로** 호출한다. 해당 없으면 건너뛴다.

## 서브에이전트 자동 호출 매핑

변경 파일/패턴을 감지해서 관련 리뷰어를 자동 호출한다.

| 트리거 조건                                                                                | 서브에이전트           | 비고               |
| ------------------------------------------------------------------------------------------ | ---------------------- | ------------------ |
| `apps/web/src/**/*.tsx` 변경 + JSX 포함                                                    | `web-a11y-reviewer`    | UI 컴포넌트 접근성 |
| `apps/mobile/src/**/*.tsx` 변경                                                            | `native-a11y-reviewer` | 9단계 이후만       |
| 변경 파일에 `useQuery`, `useMutation`, `fetch(`, `supabase.from(` 패턴 존재                | `ux-state-reviewer`    | 비동기 4상태 검사  |
| `supabase/` 변경, 또는 변경 파일에 `auth`, `RLS`, `service_role`, `.env`, `storage` 키워드 | `security-auditor`     | 보안 의미 분석     |
| `package.json` 변경 (새 의존성), 또는 변경 파일에 `import *`, 이미지 처리 코드             | `perf-budget-reviewer` | 번들/성능 검토     |

호출된 서브에이전트 결과는 verify 리포트의 "서브에이전트 리뷰 결과" 섹션에 요약 기록한다.

## 출력

검증 결과를 아래 형식으로 docs/specs/{현재단계}-verify.md에 저장해줘:

### 완료 확인 기준 결과

- [x] 또는 [ ] 각 항목별 통과/실패

### 누락 (스펙에 있는데 구현 안 됨)

- 항목 나열 (없으면 "없음")

### 스코프 크립 (구현했는데 스펙에 없음)

- 항목 나열 (없으면 "없음")

### 컨벤션 위반

- 항목 나열 (없으면 "없음")

### Codex 코드리뷰 결과

- 사용자가 제공한 Codex 리뷰 내용을 여기에 기록 (P1/P2 등급, 지적사항, 수정 여부)
- **각 지적사항은 반드시 "문제(원인)"와 "수정(대응)" 양쪽을 모두 기록한다.** 수정 후에도 원래 무엇이 문제였는지를 지우지 않는다 — 회귀 방지·이력 추적 목적.
- 형식 예:
  - **[P1] 파일:라인** — 한 줄 요약
    - 문제: 무엇이 왜 잘못되어 있었는가
    - 수정: 어떻게 고쳤는가 (✅ 수정 완료 / ⏳ 미반영)
- 없으면 "미실시" 또는 "없음"

### spec-reviewer 결과

- 복잡한 단계에서만 기록 (spec-reviewer 서브에이전트 호출 결과)
- 일치/차이/누락/스코프크립 분류별 요약
- 🔴 필수 수정 / 🟡 권장 수정 / 🟢 양호 항목 나열
- 단순한 단계에서는 "해당 없음 (단순 단계)"

### 서브에이전트 리뷰 결과

- 호출된 에이전트 목록과 각 결과 요약
- 형식 예:
  - **web-a11y-reviewer**: 🔴 1건 / 🟡 2건 / 🟢 3건 — 모달 포커스 트랩 누락
  - **ux-state-reviewer**: 4상태 완전 2/3 — ErrorMessage 컴포넌트에 error 분기 누락
  - **security-auditor**: 🟢 안전 — 민감정보 흐름 이상 없음
- 호출 조건에 해당하는 에이전트가 없으면 "해당 없음 (트리거 조건 미충족)"

### 종합 판정

- ✅ 통과 / ❌ 수정 필요 (수정 필요 시 우선순위순 정리)
