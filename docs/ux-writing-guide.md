# UX 라이팅 가이드

> 대상: 앱 안의 모든 문구 (버튼, 안내, 에러, 알림, 빈 상태)
> 함께 보기: `DESIGN.md` (시각 토큰) — 이 문서는 "무엇을 쓸까"를, 디자인 가이드는 "어떻게 보일까"를 다룬다
> 최종 수정: 2026-06-19

---

## 0. 왜 라이팅 가이드를 따로 두는가

버튼 라벨 하나, 에러 한 줄이 곧 UX다. 화면마다 말투가 달라지면 같은 앱이라는 느낌이 깨지고, 사용자는 "이 앱이 나한테 일관되게 말하고 있다"는 신뢰를 잃는다.

이 앱은 **주소·연락처 같은 민감 정보**를 받고, **이사라는 마감 있는 일정**을 다룬다. 그래서 두 가지가 특히 중요하다.

- 정보를 받을 때 **안심**시키는 말 (이 정보를 왜 받는지, 어디에 쓰는지)
- 마감이 다가올 때 **압박이 아니라 도움**으로 느껴지는 말 (D-day가 채근이 아니라 "같이 챙겨주는" 느낌)

아래 원칙은 이 두 목표를 위해 정리했다.

---

## 1. 해요체로 통일

모든 문구는 '해요체'로 쓴다. 상황·맥락 불문하고 예외 없이.

- '하십시오'체(합쇼체)나 '한다'체를 섞지 않는다
- 버튼, 에러, 알림, 약관 안내까지 전부 해요체

| 구분 | ❌ 쓰지 않음           | ✅ 해요체            |
| ---- | ---------------------- | -------------------- |
| 안내 | 날짜를 선택하십시오    | 이사 예정일을 골라요 |
| 안내 | 입력이 완료되었습니다  | 입력을 완료했어요    |
| 에러 | 네트워크 오류가 발생함 | 연결이 잠깐 끊겼어요 |

> 예외: 버튼 라벨은 명령형 동사 원형('저장하기', '다음')을 쓴다. 이건 해요체와 충돌이 아니라 버튼 고유의 관례다 (§7 참고).

---

## 2. 능동적으로 말하기

수동형보다 능동형이 짧고 명확하다. 수동형은 §10 예외에서만 쓴다.

### '되다 → 하다'

| ❌ 수동                 | ✅ 능동                 |
| ----------------------- | ----------------------- |
| 체크리스트가 생성됐어요 | 체크리스트를 만들었어요 |
| 사진이 저장됐어요       | 사진을 저장했어요       |
| 일정이 변경됐어요       | 일정을 옮겼어요         |

### '~었/됐' 빼기

| ❌                | ✅                |
| ----------------- | ----------------- |
| 완료되었습니다    | 완료했어요        |
| 항목이 추가됐어요 | 항목을 추가했어요 |

---

## 3. 긍정적으로 말하기

'안 돼요 / 없어요'로 끝내지 말고, "~하면 할 수 있어요"로 길을 열어준다.

### '없어요 → 있어요'

| ❌ 부정              | ✅ 긍정                       |
| -------------------- | ----------------------------- |
| 아직 할 일이 없어요  | 모든 일정이 순조로워요        |
| 등록된 사진이 없어요 | 첫 사진을 남겨보세요          |
| 진행한 항목이 없어요 | 첫 번째 할 일을 완료해볼까요? |

> 빈 상태(empty state)는 "비어 있음"이 아니라 "여기서 뭘 할 수 있는지"를 말한다.

### 에러 메시지 = 원인 + 해결

에러는 "뭐가 잘못됐다"로 끝내지 말고, **사용자가 뭘 하면 되는지**까지 알려준다.

| 상황          | ❌                         | ✅                                         |
| ------------- | -------------------------- | ------------------------------------------ |
| 네트워크 끊김 | 네트워크 오류              | 연결을 확인하고 다시 시도해요              |
| 서버 에러     | 알 수 없는 오류 발생       | 잠시 후 다시 시도해주세요                  |
| 입력 누락     | 입력값이 올바르지 않습니다 | 이사 예정일을 골라야 다음으로 갈 수 있어요 |

> 다이얼로그 왼쪽(보조) 버튼은 **'닫기'**로 통일한다. '취소'는 진행 중이던 작업이 취소된다고 오해할 수 있어 쓰지 않는다.

## 4. 캐주얼한 경어 — 과한 존대 빼기

'~시겠어요?', '~하시나요?', '~께' 같은 과한 경어를 쓰지 않는다. 친근하고 편한 말투가 기본이다.

| ❌ 과한 경어             | ✅ 캐주얼               |
| ------------------------ | ----------------------- |
| 이사일을 선택하시겠어요? | 이사 예정일이 언제예요? |
| 사진을 등록하시겠어요?   | 사진을 남겨볼까요?      |
| 알림을 받으시겠어요?     | 알림 받기               |

### '~시'를 뺐을 때 어색하면, 받을 정보를 주어로

사용자 정보를 받는 질문에서 기계적으로 '~시'를 빼면 어색해진다. 이때는 **알고 싶은 정보를 주어로** 써서 문장을 다시 만든다.

| ❌ 어색           | ✅ 정보를 주어로              |
| ----------------- | ----------------------------- |
| 언제 이사 가요?   | 이사 예정일이 언제예요?       |
| 어떤 집에 살아요? | 지금 사는 집은 어떤 형태예요? |

## 5. '{명사} + {명사}' 풀어쓰기

한자어 명사를 동사로 풀면 더 부드럽고 캐주얼해진다.

| ❌ 딱딱          | ✅ 풀어쓰기                  |
| ---------------- | ---------------------------- |
| 일정 재배치 완료 | 남은 할 일을 다시 배치했어요 |
| 항목 완료 처리   | 할 일을 끝냈어요             |
| 계정 삭제 진행   | 계정을 삭제할게요            |

풀어쓰기 어려우면 '{명사}가 {명사}해서' 형태로만 풀어도 한결 낫다. 예: "연결이 불안정해서 다시 시도해요."

---

## 6. 한 줄로 말하기

모바일 화면은 좁다. 핵심을 한 줄(이상적으로는 한 호흡)에 담는다.

- 가장 중요한 메시지를 **맨 앞**에
- 부연 설명은 다음 줄로 분리하거나 작게
- '되어요'는 모두 '돼요'로 통일 (좁은 공간 고려)

| ❌ 길게                                                                                    | ✅ 한 줄                               |
| ------------------------------------------------------------------------------------------ | -------------------------------------- |
| 입주하실 때 사진을 미리 찍어두시면 나중에 보증금 관련 분쟁이 생겼을 때 도움이 될 수 있어요 | 입주 사진을 찍어두면 보증금을 지켜줘요 |

---

## 7. 버튼 작성법

버튼은 **그 버튼을 누르면 무슨 일이 일어나는지**를 동사로 말한다. 추상적인 '확인 / 완료'를 피하고 구체적인 행동을 쓴다.

### 원칙

- 동사 원형 + '하기' 형태가 기본: '저장하기', '다음', '맞춤 체크리스트 만들기'
- 버튼만 보고도 결과를 예측할 수 있어야 한다
- 화면에서 가장 중요한 행동 1개 = 메인(primary) 버튼

| 상황          | ❌ 모호 | ✅ 구체적              |
| ------------- | ------- | ---------------------- |
| 온보딩 마지막 | 완료    | 맞춤 체크리스트 만들기 |
| 사진 저장     | 확인    | 사진 저장하기          |
| 알림 권한     | 확인    | 알림 받기              |
| 프리체크      | 제출    | N개 체크 완료          |

### 파괴적 행동 버튼

삭제처럼 되돌릴 수 없는 행동은 버튼 라벨에 **무엇이 사라지는지** 명시한다.

| ❌   | ✅            |
| ---- | ------------- |
| 확인 | 계정 삭제하기 |
| 예   | 사진 삭제하기 |

> 다이얼로그 구성: 왼쪽 '닫기' / 오른쪽 '삭제하기'(danger 스타일). 파괴적 확인은 OS 네이티브 다이얼로그 표준을 따른다.

## 8. 안심시키기 (이 앱의 핵심)

이 앱은 주소·연락처·이사 일정 같은 민감 정보를 받는다. 정보를 요청하는 화면에서는 **왜 받는지 + 어디에 쓰는지**를 함께 말해 사용자를 안심시킨다.

### 정보를 받을 때

| ❌ 그냥 요구        | ✅ 안심                                           |
| ------------------- | ------------------------------------------------- |
| 주소를 입력하세요   | 입력한 주소는 일정 안내에만 써요                  |
| 연락처를 입력하세요 | 연락처는 견적 비교에만 쓰고, 따로 저장하지 않아요 |

### 권한을 요청할 때

권한 요청(알림 등)은 "받으면 뭐가 좋은지 + 스팸은 없다"를 말한다.

- ✅ 제목: 이사, 깜빡하지 않게 챙겨드릴게요
- ✅ 본문: 할 일 있는 날 아침에 딱 한 번, 필요한 것만 알려드려요. 광고·스팸은 없어요.

> 민감한 정보 수집 안내에서는 사용자를 안심시키기 위해 수동형을 써도 된다 (§10 예외). 예: "입력한 정보는 암호화되어 보관돼요."

## 9. 마감을 부드럽게 (D-day 카피)

이사는 마감이 있는 일정이다. D-day 알림이 **채근**이 아니라 **같이 챙겨주는 느낌**이 되도록 쓴다. 압박형 부정문 대신 도움형 긍정문을 쓴다.

| 시점  | ❌ 압박형               | ✅ 도움형                                       |
| ----- | ----------------------- | ----------------------------------------------- |
| D-7   | 일주일밖에 안 남았어요  | 지금부터 하나씩 챙기면 여유로워요               |
| D-3   | 아직도 안 끝냈어요?     | 오늘 할 일 N개만 확인해요                       |
| D-1   | 내일인데 준비 안 했어요 | 마지막 점검, 빠진 거 없는지 확인해요            |
| D-day | 오늘까지 다 했어야 해요 | 오늘 이사 가는 날! 마지막 체크리스트만 확인해요 |

### 진행률 응원

완료율에 따라 격려한다. 못 한 걸 지적하지 않고, 한 걸 인정한다.

| 진행률 | ✅                              |
| ------ | ------------------------------- |
| 0%     | 첫 번째 할 일을 완료해볼까요?   |
| 1~50%  | 좋은 시작이에요! 계속 가볼까요? |
| 51~80% | 절반 넘었어요! 거의 다 왔어요   |
| 81~99% | 마지막 N개만 남았어요!          |
| 100%   | 모든 할 일을 완료했어요!        |

### 밀린 항목 재배치

밀린 일을 "실패"가 아니라 "다시 정리해줄게"로 표현한다.

- ❌ N개를 놓쳤어요
- ✅ 남은 할 일을 다시 배치했어요 / 지금 해도 괜찮아요

> 이모지는 D-day 마일스톤 같은 특별한 순간에만 1개. 본문 곳곳에 흩뿌리지 않는다.

## 10. 예외 규칙

원칙을 깨도 더 명확해지는 경우가 있다. 이때만 예외를 둔다.

### 수동형을 써도 되는 경우

- **사용자에게 미치는 영향을 알려줄 때** (연체·해지·적용 등): 인과관계를 분명히 한다. 예: "지금 삭제하면 모든 데이터가 함께 삭제돼요."
- **민감 정보 안내로 안심시킬 때**: 예: "입력한 정보는 암호화되어 보관돼요."
- **기간 만료·종료를 강조할 때**: 주어(기간·서비스)를 강조하면 뉘앙스가 정확해진다.

### 경어(시나요?/셨나요?)를 써도 되는 경우

- **사용자 상황을 추정해 정중히 물을 때**: 앱이 확실히 모르는 정보를 사용자에게 판단하게 할 때. 예: "이사 일정이 바뀌셨나요?"
- **사용자의 선의가 필요할 때**: 설문·피드백 요청처럼.

### 부정형을 써도 되는 경우

- **정책상 쓸 수 없는 기능을 명확히 알릴 때**: 이유를 함께 안내한다.
- **일부 기능만 막혔을 때**: 어떤 기능이 안 되는지 분명히 한다. 예: "사진은 로그인해야 저장할 수 있어요."

## 11. 빠른 체크리스트

문구를 쓰거나 리뷰할 때 훑어본다.

- [ ] 해요체인가? (버튼 라벨 제외)
- [ ] 능동형인가? ('되다' → '하다')
- [ ] 긍정형인가? ('없어요' → '있어요/해볼까요')
- [ ] 과한 경어를 뺐는가? ('~하시겠어요' → '~할까요')
- [ ] 한 줄에 핵심이 담겼는가? ('되어요' → '돼요')
- [ ] 버튼이 결과를 구체적으로 말하는가? ('확인' → '저장하기')
- [ ] 민감 정보 화면에 안심 문구가 있는가?
- [ ] D-day 문구가 압박이 아니라 도움인가?
- [ ] (예외라면) §10에 해당하는 정당한 이유가 있는가?

## 부록: 우리 말투 3줄 요약

1. **친근하게, 해요체로** — 과한 존대 없이 옆에서 챙겨주는 친구처럼
2. **안심시키며** — 민감한 정보를 받을 땐 왜·어디에 쓰는지 먼저 말한다
3. **압박 대신 응원** — 마감이 다가와도 채근하지 않고 같이 챙긴다
